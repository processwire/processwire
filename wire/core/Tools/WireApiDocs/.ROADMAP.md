1. Properties — the biggest gap
   getMethods() covers methods, but ProcessWire's API is heavily property-based ($page->title, $page->images, etc.). Many core classes document these via @property and @property-read tags
   in the class phpdoc header. A getProperties($class) that parses those tags would make the API surface much more complete — right now there's no programmatic way to ask "what properties
   does Page expose?"

2. Hookable flag missing from getMethods() output
   The method strips ___ prefixes and returns hookable methods under their public names, but the returned array doesn't indicate which ones are hookable. Adding a 'hookable' => true/false
   key to each entry would let callers (and agents) know which methods are extension points without having to re-inspect the source.

3. Constants
   Class constants aren't covered at all. Inputfield has 15+ collapsed/skipLabel constants, Page has STATUS_* constants — these are important parts of the API. A getConstants($class) via
   ReflectionClass::getConstants() would fill that gap.

4. Class info
   No way to ask about a class's inheritance chain, implemented interfaces, or traits. A getClassInfo($class) returning parent, interfaces, and traits would help agents understand where
   inherited behavior comes from (e.g., "why does Pages have a wire() method?").

5. Content search
   Currently requires knowing the exact class name. A search($query) that looks across class names and API.md content would let agents ask "what class handles image resizing?" rather than
   having to enumerate all classes.
